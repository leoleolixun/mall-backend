# go-mall 后续开发与验收总路线

> 基线日期：2026-07-16
>
> 适用仓库：`mall-backend`、`mall-frontend/mall-pc-web`、`mall-frontend/mall-admin`、`mall-frontend/mall-uniapp`

## 1. 当前基线

### 已有可运行系统

- Go 后端：Gin、GORM、MySQL、Redis、JWT、Swagger/OpenAPI。
- 买家 PC Web：认证、商品、购物车、结算、订单、支付宝支付入口、优惠券、收藏和售后页面已有真实接口接入。
- 商家后台：商品、分类、SKU、库存、订单、发货、售后、优惠券、员工、角色、顾客、经营概览，以及店主/管理员只读结算单和结算账本已有真实接口接入。
- 部署：后端、PC Web、商家后台都通过 GitHub Actions 发布；只有手动触发或提交信息包含 `[deploy]` 才执行测试、构建和部署。
- 后端定时任务：超时未支付订单取消、已发货订单自动完成、退款结果对账。
- UniApp：账号密码下的 H5/小程序交易闭环、跨商户交易结算与交易级支付宝 H5 支付、CLI 构建、契约测试、H5 三浏览器验收和独立 CI/CD 已完成，等待部署、隔离写环境和小程序真机补证。

### 尚未完成或尚未真实验收

- 订单明细优惠券分摊、舍入尾差、实付金额快照和历史数据回填已经完成。
- 图片上传代码已存在，但腾讯云 COS 或七牛云 Kodo 的真实配置与完整业务联调未完成。
- 支付宝 PC 网页支付、异步通知、主动查询和真实退款需要在沙箱或生产环境做完整验收。

## 2. 第一版范围

第一版定义为“单商户 PC 网页商城可真实交易版本”。只有下列闭环必须通过：

```text
商家登录
  -> 创建分类、商品和 SKU
  -> 上传图片
  -> 调整库存并上架
  -> 买家注册或账号密码登录
  -> 浏览商品、加购物车、领取优惠券
  -> 预览并创建订单
  -> 支付宝网页支付
  -> 商家发货
  -> 买家确认收货
  -> 买家申请售后
  -> 商家审核并退款
  -> 系统主动查询和定时对账
```

第一版明确不包含：

- 微信登录和微信支付。
- H5、小程序和 App。
- 多商户购物车、拆单、结算和平台抽佣。
- 平台运营后台。
- 第三方快递实时轨迹。
- 商品评价、预售、套餐、对比和到店自提。

没有实现的入口应从正式导航中隐藏或明确显示“暂未开放”，不能调用伪接口或返回伪成功。

## 4. 每个功能统一开发过程

以后每个功能都按以下顺序开发，不直接从页面或数据库表开始写。

### 4.1 明确需求和边界

先写清楚：

- 谁可以执行，买家、商家老板、销售、库管还是平台管理员。
- 前置状态和允许的状态流转。
- 输入、输出、错误码和幂等要求。
- 会修改哪些数据，是否必须使用事务或行锁。
- 本期明确不处理什么。

### 4.2 先更新 OpenAPI

开发接口前先在 `docs/openapi.yaml` 中完成：

- path、method、鉴权方式和权限。
- request schema、response schema 和示例。
- 正常响应以及 400、401、403、404、409、500 响应。
- 当前接口标记为“已实现”还是“规划中”。

前端根据 OpenAPI 并行定义 TypeScript 类型和 Mock；后端实现不能随意偏离契约。契约变化必须先修改文档并通知前端。

### 4.3 设计数据变化

涉及数据库时先写数据设计：

- 表、字段、类型、默认值、非空约束和索引。
- 唯一约束能否作为并发兜底。
- 老数据如何回填。
- 是否支持旧版本代码和新版本代码短时间共存。
- 出错后的补偿、对账或人工处理方式。

生产迁移必须向前兼容。先加可空字段并回填，再切换代码，最后才考虑收紧约束或删除旧字段。

### 4.4 按后端分层实现

推荐顺序：

```text
Model / Migration
  -> DTO
  -> Repository
  -> Service
  -> Handler
  -> Router
  -> main.go 依赖装配
  -> OpenAPI 实现状态更新
```

职责保持清楚：

- Handler：HTTP 参数、当前用户、状态码和统一响应。
- Service：业务校验、事务、幂等、权限范围和状态机。
- Repository：数据库查询和持久化，不处理 Gin。
- 外部网关：支付宝、对象存储等第三方调用单独封装并可替换为测试实现。

### 4.5 编写测试

最低测试要求：

- 正常路径。
- 参数边界。
- 未登录和无权限。
- 数据不存在和跨用户、跨商户访问。
- 重复请求和并发请求。
- 事务中途失败时不能留下半成品数据。
- 第三方成功、失败、超时和结果未知。

资金、库存、优惠券和权限相关改动必须有自动化测试，不能只依赖浏览器手工点击。

### 4.6 前端接入

前端接入顺序：

```text
API 类型和 client
  -> query/mutation 状态
  -> 页面和表单
  -> loading/empty/error/disabled 状态
  -> 操作成功后的缓存刷新
  -> 权限和路由保护
```

前端不能根据按钮是否可见代替后端权限校验。金额使用“分”为接口单位，只在展示层转换为元。

### 4.7 开发者自测

开发者先完成：

- 后端目标包测试。
- 前端 TypeScript 检查。
- Swagger 请求验证。
- 浏览器主流程验证。
- 数据库关键字段前后对比。
- 第三方请求 ID、业务单号和日志检查。

### 4.8 集成环境验收

使用独立测试账号、测试商户、测试数据库或至少独立数据库 schema。Redis 使用独立 DB 或 key 前缀，不能在生产数据上运行会创建、删除和退款的验收脚本。

### 4.9 提交和发布

- 后端、前端分别提交，提交信息遵循仓库规范。
- 普通推送不加 `[deploy]`，不会执行部署测试。
- 需要发布时，最后一个提交信息加 `[deploy]`，或在 GitHub 手动运行 workflow。
- 发布提交必须能说明需求、迁移、配置变化和回滚方式。

### 4.10 发布后验收

发布后必须检查：

- `/health`、Swagger 和 OpenAPI。
- 登录、商品列表和一个核心写接口。
- systemd 服务、定时器和最近日志。
- Nginx、证书和前端静态资源。
- 数据库连接数、Redis、第三方回调和错误率。
- 旧版本回滚是否仍兼容本次数据库结构。

## 6. M1：订单优惠分摊与退款金额一致性

这是当前最重要的业务正确性任务。订单已经记录总优惠，但 `order_items` 还没有保存每项承担的优惠金额，后续退款会因为临时计算和舍入产生误差。

### 数据设计

给订单明细增加至少两个金额快照：

```text
discount_amount  当前明细承担的优惠金额，单位分
payable_amount   当前明细优惠后的实付商品金额，单位分
```

运费继续保存在订单主表，第一版售后只退商品实付金额，不自动退运费。

当前售后请求没有退款数量字段，因此 V1 按“整个订单明细”申请退款。按件部分退款属于后续功能，不能用修改退款金额的方式绕过数量和累计退款校验。

### 分摊规则

建议采用确定性规则：

```text
前 N-1 项优惠 = floor(订单优惠 * 明细小计 / 商品总额)
最后一项优惠 = 订单优惠 - 前面所有明细优惠之和
明细实付 = 明细小计 - 明细优惠
```

“最后一项”必须按稳定顺序确定，例如 `order_items.id` 或请求中的固定排序，不能依赖 map 遍历顺序。

必须满足：

```text
sum(order_items.subtotal) = orders.goods_amount
sum(order_items.discount_amount) = orders.discount_amount
sum(order_items.payable_amount) = orders.goods_amount - orders.discount_amount
orders.payable_amount = sum(order_items.payable_amount) + orders.freight_amount
```

### 开发过程

1. 修改 `OrderItem` 模型和迁移。
2. 为历史订单设计回填逻辑；回填算法必须与新订单一致。
3. 把分摊算法写成无数据库依赖的纯函数并单独测试。
4. 订单预览返回每项优惠和实付金额。
5. 创建订单时在事务内保存分摊快照。
6. 售后申请直接读取订单明细实付金额，不再临时按订单比例计算。
7. 退款累计金额不能超过订单已支付金额，也不能超过该明细尚未退款金额。
8. 同步修改 DTO、OpenAPI、PC Web 订单明细和商家后台售后金额展示。

### 自动化验收

- 无优惠、单商品、多商品、多数量均计算正确。
- 100 分优惠分给 3 个同价商品时，总和仍精确等于 100 分。
- 商品金额不能出现负数，优惠不能大于商品总额。
- 重复创建同一幂等订单不会重复核销优惠券或扣库存。
- 同一订单明细不能同时存在两个进行中的售后；已拒绝或已取消后可以重新申请。
- 并发批准退款不会让累计退款超过支付金额。
- 事务任一步骤失败时，订单、订单项、库存、购物车和优惠券状态同时回滚。

### 手工验收

创建一张包含 3 个不同价格 SKU 和 1 张优惠券的订单，记录：

- 预览金额。
- 订单主表金额。
- 每个订单明细分摊金额。
- 支付宝实付金额。
- 每个明细可退款金额。

所有金额等式一致，退款后支付宝和本地累计退款金额一致，M1 才通过。

## 7. M2：对象存储真实联调

第一版只选择一个供应商作为正式实现，推荐在腾讯云 COS 和七牛云 Kodo 中选一个。另一家保留接口扩展能力，不要求同时上线。

### 开发过程

1. 在服务器 `config.yaml` 填写供应商、region、bucket、domain、access key 等配置。
2. 密钥只存服务器，不提交 GitHub；`config.example.yaml` 只保留字段和假值。
3. 启动时校验对象存储配置，生产环境禁止使用本地文件存储或空密钥。
4. 验证 `/api/v1/uploads` 和 `/api/v1/merchant/uploads` 的鉴权、场景和权限。
5. 接入用户头像、商品封面、SKU 图片和售后凭证。
6. 定义对象 key：环境、业务场景、年月和随机文件名，不直接使用用户原文件名。
7. 定义图片大小、MIME、扩展名、数量和分辨率限制。
8. 明确替换图片和删除业务数据时的垃圾文件清理策略；第一版可以异步延迟清理，但必须有记录。

### 验收过程

- 未登录上传返回 401；没有 `upload:write` 权限的员工返回 403。
- JPG、PNG、WebP 合法图片成功；伪装扩展名、超大文件和非图片失败。
- 上传返回 HTTPS URL，PC Web、商家后台和手机浏览器均可访问。
- 商品图可以保存、重新查询并展示；头像刷新页面后仍存在。
- 售后多图顺序和数量正确，商家后台可查看。
- 日志不打印 SecretKey、Authorization 或完整私钥。
- 对象 key 不冲突，重复文件名不会覆盖已有文件。

## 8. M3：支付宝真实支付与退款

先在支付宝沙箱完成全流程，再使用生产环境一笔最小金额订单验证。不能直接用大额生产订单测试。

### 支付开发与验收

1. 校验应用 ID、应用私钥、支付宝公钥、网关、回调地址和返回地址。
2. 创建订单后请求 `POST /api/v1/payments`，只允许 `alipay + page`。
3. 相同订单重复请求返回同一待支付单；切换渠道或场景明确拒绝。
4. 浏览器跳转支付宝，完成支付后进入结果页。
5. 返回页只用于展示，最终结果以异步通知或主动查询为准。
6. 回调必须验签，并校验 app_id、订单号、金额、seller_id（配置后）和交易状态。
7. 回调重复到达时幂等成功，不能重复改库存或重复写支付结果。
8. 前端轮询支付详情并在必要时调用主动同步，最终显示成功、失败或仍处理中。

支付通过条件：

- 支付宝交易金额 = `payments.amount` = `orders.payable_amount`。
- 支付成功后订单与支付单同时进入正确状态，并记录交易号和支付时间。
- 支付中关闭页面，重新进入订单页仍能继续查询结果。
- 伪造签名、错误金额和错误订单号均不能更新本地状态。
- 已取消、已支付订单不能再次创建有效支付意图。

### 退款开发与验收

1. 买家对已支付、已发货或已完成订单明细发起售后。
2. 商家批准时创建唯一退款单，并使用固定 `refund_no` 调用支付宝。
3. 明确成功时落库成功；明确失败时记录失败原因；超时或网络错误进入 `unknown`，不能直接判失败。
4. 商家可以主动同步退款结果，定时任务也会按 `next_retry_at` 对账。
5. 相同退款号重复调用保持幂等，不生成第二笔退款。

退款通过条件：

- 本地退款金额不超过明细实付金额和订单剩余可退金额。
- 支付宝退款金额、退款号、交易号与本地一致。
- 网络超时后重试不会重复退款。
- 退款最终成功后，Refund、AfterSale 和 Payment 的状态保持一致。
- 定时对账任务失败可观察，成功后不再重复扫描已完成退款。

### 需要保存的验收证据

- 支付宝交易号、退款号和请求时间。

### 发布前

1. 确认提交范围和数据库变更。
2. 备份生产数据库并记录备份文件、时间和校验结果。
3. 在服务器运行配置检查。
4. 确认 MySQL、Redis、支付宝、对象存储和磁盘正常。
5. 确认旧二进制或旧前端压缩包可用于回滚。

### 发布中

1. 使用带 `[deploy]` 的提交或手动 workflow 发布后端。
2. GitHub Actions 执行测试、vet、构建、配置检查、迁移、服务重启和 `/health` 检查。
3. 后端成功后再发布商家后台和 PC Web，避免新前端先调用尚未部署的接口。
4. 数据库迁移采用向前兼容方式，不能依赖简单回滚 schema。

### 发布后 30 分钟

- 检查 `systemctl status go-mall`。
- 检查三个定时器状态和最近执行结果。
- 检查 Nginx 5xx、Go panic、数据库错误和 Redis 错误。
- 运行只读烟雾测试，再运行一笔受控写流程。
- 验证 Swagger、PC Web、商家后台和支付宝回调地址。

### 回滚条件

出现以下任一情况立即停止流量或回滚：

- 支付成功但订单无法变为已支付。
- 重复扣款、重复退款、超卖或跨商户数据泄露。
- 持续 5xx、服务反复重启或迁移破坏历史数据。
- 登录全面不可用或关键配置泄露。

回滚代码后仍需处理已经发生的数据差异。资金和库存问题不能只靠回滚二进制，必须冻结相关操作并执行对账或补偿。

## 15. M8：多商户

多商户不能只给现有表增加 `merchant_id`。编码前先完成领域设计和迁移方案。

> 当前进度（2026-07-16）：领域与迁移方案已固化在 `m8-multi-merchant-domain-design.md`。M8-A 公共目录、M8-B 版本化迁移/历史回填、M8-C 原子拆单以及 M8-D 交易级支付、退款冲正和结算台账均已完成工程实现。M8-B 至 M8-D 已在临时 MySQL 8.4 完成迁移、回填、事务故障、并发、支付分配、退款和结算验收；生产备份副本、生产 HTTP 写验收及真实渠道资金仍需补证。

M8-C 的拆单证据记录在 `m8-c-trade-acceptance-2026-07-16.md`；M8-D 的支付、退款、结算及三端证据记录在 `m8-d-payment-settlement-acceptance-2026-07-16.md`。这些记录只代表工程与隔离环境通过，不代表生产库已经迁移或真实资金链路已经验收。

### 设计阶段

至少确定：

- 购物车是否允许同时包含多个商户商品。
- 一次结算是否生成交易单和多个商户子订单。
- 地址、运费、优惠券、支付、退款、发货和售后如何按子订单处理。
- 平台优惠券与商户优惠券的承担方。
- 平台抽佣、商户应结算金额、退款冲正和结算周期。
- 商户停用、商品下架和账号禁用后的处理。
- 订单号、支付单号、退款号、结算单号和对账单关系。

推荐模型：

```text
checkout / trade
  -> merchant_orders（每商户一张）
  -> order_items
  -> payment（一次合并支付或每商户支付，先明确方案）
  -> payment_allocations（支付金额分配）
  -> refunds / refund_allocations
  -> settlements / settlement_items
```

### 实现顺序

1. 公共商户信息和商户商品列表。（已完成）
2. 购物车按商户分组所需的后端数据。（已完成，PC Web 和 UniApp 已适配）
3. 多商户订单预览和运费计算。（已完成）
4. 事务内创建交易单和子订单。（已完成）
5. 支付金额分配和回调。（已完成工程验收）
6. 商户独立发货、售后和退款。（已完成工程验收）
7. 结算台账、对账和冲正。（已完成工程验收；实际打款待 M9/合规分账产品）
8. PC/H5/小程序和商家后台适配。（已完成工程验收）

### 核心验收

- 任意商户账号都不能查询或修改其他商户数据。
- 子订单商品金额、优惠、运费和支付分配总和等于交易单金额。
- 一个商户库存不足时，整次提交失败还是只失败该商户必须符合既定规则。
- 一个子订单退款不改变其他子订单状态和结算金额。
- 平台收入、商户应收、退款和实际渠道资金可以逐笔对账。
- 100 个并发跨商户下单不会超卖、重复拆单或重复支付。

## 16. M9：平台运营后台

平台后台与商家后台是不同权限域，使用独立登录、JWT、路由前缀和审计日志。

### 开发顺序

1. 平台管理员登录、角色、权限和操作审计。
2. 商户入驻、详情、启用、停用和配置。
3. 用户、订单、售后和支付的跨商户只读查询。
4. 风险操作：冻结商户、关闭商品、人工备注和异常单处理。
5. 平台经营概览、结算和对账。
6. 配置和操作审批，避免高风险操作由单人无记录执行。

### 验收过程

- 商家 Token 不能访问 `/admin`，平台 Token 也不能冒充商家写业务数据。
- 所有跨商户查看和高风险写操作都有审计记录。
- 停用商户后，买家不能新下单，但历史订单、退款和结算仍可处理。
- 平台查询支持分页和必要索引，不产生全表慢查询。
- 敏感数据按角色脱敏，导出和批量操作受权限控制。

## 17. M10：后续增强

按真实业务需求逐项立项，不同时开发：

- 第三方快递轨迹：优先接聚合服务，保留商家自配送状态；接入前继续只展示快递公司和单号。
- 商品评价：仅已完成订单可评价，防重复、审核、统计和图片存储。
- 预售：独立库存、定金/尾款、发货时间和取消规则。
- 套餐和组合购：套餐价格快照、库存拆解和退款分摊。
- 到店自提：门店、核销码、核销权限和过期处理。
- 手机号绑定：短信服务、验证码限流、唯一性和账号合并规则。
- 搜索：数据量增加后再评估 MySQL 全文索引或 Elasticsearch。
- 消息通知：订单状态、发货、退款和库存预警，使用可靠任务或消息队列。

每项都继续使用第 4 节的统一开发过程和独立验收门禁。

## 20. 缺陷等级与放行规则

| 等级 | 示例 | 放行规则 |
| --- | --- | --- |
| P0 | 重复扣款、退款错误、超卖、越权、数据丢失、密钥泄露 | 必须修复，禁止上线 |
| P1 | 主交易流程无法完成、页面持续报错、核心状态错误 | 必须修复，禁止上线 |
| P2 | 次要功能错误、有明确替代路径 | 评估后可延期，必须记录 |
| P3 | 文案、轻微样式或低频体验问题 | 可进入后续迭代 |

一个阶段只有在以下条件全部满足时才算完成：

- 代码、OpenAPI、配置模板和迁移一致。
- 正常、异常、幂等、并发和权限用例通过。
- 前后端在集成环境完成真实联调。
- 验收证据可追踪到提交、数据和日志。
- 没有 P0/P1 缺陷。
- 已明确部署、监控和回滚方式。
