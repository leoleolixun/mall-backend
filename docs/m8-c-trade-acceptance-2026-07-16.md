## 1. 验收范围

本轮实现并验收以下交易接口：

```http
POST /api/v1/trades/preview
POST /api/v1/trades
GET  /api/v1/trades
GET  /api/v1/trades/{id}
POST /api/v1/trades/{id}/cancel
```

覆盖的业务边界：

- 一个购物车可同时选择多个启用商户的 SKU。
- 一次预览按商户计算商品金额、商户券优惠、运费和应付金额，并汇总交易总额。
- 一次创建在一个 MySQL 事务中生成一张交易单和每商户一张子订单。
- 任一商户、商品、SKU、库存、价格、数量、地址或商户券校验失败时整笔失败。
- 相同用户和幂等令牌重复提交只返回同一张交易，不重复扣库存或核销券。
- 未支付交易只能整笔取消，统一恢复所有子订单库存和商户券。
- 旧 `/orders` 继续作为默认商户单订单兼容入口，不能用于跨商户拆单或交易支付。

## 2. 前端行为

PC Web 和 UniApp 已完成：

- 购物车按商户稳定分组，并显示商户名称。
- 每个商户分组独立选择本商户优惠券。
- 结算页显示各商户商品金额、优惠、运费、应付金额和交易合计。
- 提交前立即重新请求交易预览，只使用最新 `idempotency_token` 创建交易。
- 创建后按一张或多张子订单进入订单视图。
- M8-D 完成前不显示多商户交易支付按钮，不拿第一张子订单调用旧支付接口。

## 4. 隔离 MySQL 验收

在临时 MySQL 8.4 环境执行真实事务集成测试：

```bash
MALL_ALLOW_TRADE_INTEGRATION=1 \
MALL_TRADE_TEST_DSN='<isolated-test-dsn>' \
go test -count=1 -run TestMultiMerchantTradeIntegrationOnMySQL -v ./internal/service
```

通过场景：

1. 两个商户、两张商户券生成一张交易和两张子订单，查询、幂等和整笔取消正确。
2. 券与商户不匹配时拒绝创建，没有部分写入。
3. 预览后价格变化时要求重新预览，没有部分写入。
4. 第二个商户写入故障时，第一个商户的订单、库存和券核销全部回滚。
5. 100 次并发使用同一幂等令牌只创建一张交易。
6. 100 个用户竞争同一 SKU 的 10 件库存时成功数量不超过 10，库存不为负。

高并发场景中的 `SELECT ... FOR UPDATE` 出现约 200-300 ms 等锁时间，属于测试刻意制造的行锁竞争；未出现死锁、超卖或残留部分交易。

## 6. 下一阶段准入

M8-D 开发必须继续遵守：

- 一张交易只能有一张有效待支付单，数据库唯一键作为并发兜底。
- 支付金额等于交易应付金额，所有支付分配之和等于支付金额。
- 每张子订单的支付分配等于该子订单应付金额，不能把第一张子订单伪装成整笔支付。
- 退款只能冲减原子订单支付分配的剩余可退金额。
- 佣金、退款冲正和结算使用可审计的不可变流水。
